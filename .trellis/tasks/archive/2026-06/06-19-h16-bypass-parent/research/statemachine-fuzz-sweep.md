# Attack-Surface Sweep: State-Machine Fuzz + Error-Differential Oracle

- **Query**: 逆向客户端发非法局内动作 → 服务端错误响应差异 → 反推对手隐藏手牌 (oracle attack)
- **Scope**: 纯离线设计 + 静态分析 (无发包/无联网)
- **Date**: 2026-06-20
- **Posture**: assumption-first。本文不写"行不通"作为开场;而是严格区分"错误差异泄漏对手隐藏牌(真 oracle)" vs "只是拒绝你自己的非法操作(无 oracle 价值)",并把每个候选动作放进判定表。

---

## 0. 攻击模型 (用户原话提炼)

> "不该你出牌的时候出牌,服务器有时返回错误、有时状态机直接重置。看错误差异化,如果有明显差异就可能有问题,可能分析出卡牌。"

形式化为 **error-differential oracle**:

> 设 H = 对手隐藏手牌(真值)。我发一个非法/越权动作 A。观测服务端响应 R(A)。
> **当且仅当 R(A) 是 H 的函数 (R 随 H 变化)** 时,A 构成一个能泄漏对手手牌的 oracle。
> 若 R(A) 只是我自己状态/我自己手牌的函数,则它只是"拒绝我的非法操作",**零 oracle 价值**。

这是本任务成败的全部核心。下面 §1 对每个动作做这个判定。

## 1. 逐动作 Oracle 判定表 (最关键 §)

### 1.1 静态协议事实 (已读源码确认,别重测)

来自 `apk_research/decrypted-lua/app/Protocols/GameProtocolGT.lua` 和 `stable/protocol.py`:

**两层动作彻底区分:**

| 层 | 协议 | XY_ID / sub_cmd | 响应 | 静态可见性 |
|---|---|---|---|---|
| 房间级 | `ReqPlayerAct` | XY_ID=11016 (CMDT_REQPLAYERACT) | `RespPlayerAct.flag` = SUCCESS(0) / SHOW_MESSAGE(1+msgbox) | **完整** (lua L147-233) |
| 局内牌动作 | game-event 0x2BC0 + player_action | sub_cmd 0x021B discard / 0x021F meld / 0x0220 win;请求走 0x2C2E player_action,响应 0x2C2F action_rsp | 响应处理在热更牌局场景,**静态包缺失** | **缺失** |

**房间级 ReqPlayerAct.ACTION 枚举** (lua L151-163):
`NONE=0 / SITDOWN=1 / STANDUP=2 / READY=3 / SEEGAME=4 / CHANGETOSTART=5 / CHANGETOSEEGAME=6 / CHANGESEAT=7 / LEAVEROOM=8 / SEEGAME2=9`

**响应只有两种 flag** (lua L194-198): `SUCCESS=0` 或 `SHOW_MESSAGE=1`(带 MsgBox 文本)。MsgBox 字段 = type/caption/text/iconbtn/delay/colour/web/action。

**关键决定性事实 — `action_notify (0x0016)` 是服务端 PROACTIVE 预告:**
`stable/protocol.py:392` + `_extract_optional_action()` (L596-622):服务端在轮到你可以吃/碰/杠/胡/过时,**主动**下发一个 `optional_action` 帧,bitmask = `chi=0x01 / pon=0x02 / kong=0x04 / hu=0x08 / pass=0x10`,告诉你**当前合法的动作集合**。
→ 这意味着:你声明吃/碰/杠/胡是否合法,**服务端已经替你算好并下发了**。你不需要靠"发非法动作看报错"来探测自己能不能吃——服务端白送。这同时说明:这些校验是**针对请求者自己的手牌**做的(你的牌你已知)。

### 1.2 判定表

判定标准:**响应 R(A) 是否是对手隐藏手牌 H 的函数?**

| # | 动作 A | 服务端校验逻辑读取什么? | R(A) 是 H 的函数? | Oracle 价值 | 备注 |
|---|---|---|---|---|---|
| R1 | 房间级 ReqPlayerAct (SITDOWN/READY/SEEGAME/CHANGESEAT/LEAVEROOM…) 错状态发 | 你自己的座位/房间状态 | **否** | **零** | flag/msgbox 只反映你自己合法性,与对手手牌无关。最多泄漏房间元数据(座位占用),非手牌 |
| R2 | discard (0x021B) 不该你出牌时出牌 | 当前轮到谁 (公开的轮次状态) | **否** | **零** | 轮次是公开信息;拒绝只说"没轮到你" |
| R3 | discard 出一张不在你手里的牌 | 你自己的手牌 | **否** | **零** | 校验你自己的牌。泄漏的是你已知信息 |
| R4 | discard 重复出牌 / 跳轮出牌 | 你自己 + 轮次状态 | **否** | **零** | 同上 |
| R5 | 声明 chi (吃) | 上家打的牌(公开) + **你自己**手里相邻两张 | **否** | **零** | 校验你自己的牌。且 action_notify 已预告你能否吃 |
| R6 | 声明 pon/kong (碰/杠) | **你自己**有无对子/刻子 | **否** | **零** | 校验你自己的牌。action_notify 已预告 |
| R7 | 声明 win/hu (自摸/点炮胡) | **你自己**手牌 + 这张牌能否和 | **否** | **零** | 校验你自己的牌型。action_notify 已预告 hu 位 |
| **R8** | **抢杠胡 (rob-the-kong)** | 别人**加杠的那张牌**(加杠是公开广播的 meld) + **你自己**手牌能否胡那张 | **否** | **零** | ⚠ 直觉以为依赖对手,实则:被杠的牌已公开(meld 0x021F),校验的是**你**能否胡这张公开牌。仍是你自己的牌 |
| **R9** | **局末查叫/查听 (流局时校验各家听牌)** | **所有玩家的手牌**(含对手隐藏牌)——服务端必须读 H 才能判定各家是否听牌、是否相公、退税 | **是 (理论上)** | **潜在真 oracle,但** | 见 §1.3 详析。**最强候选**,但被 H16 全局过滤墙挡住 |
| R10 | 一炮多响 (多家同时胡你点的炮) | 多个**其他**玩家手牌 | **是 (理论上)** | **潜在,同 R9** | 服务端要判定哪几家能胡你这张。但结算帧 (win 0x0220) 实证 = 23B 纯数值,不亮牌 |
| R11 | 报听/听牌广播 | 报听者自己手牌 | **否** | **零** | 是你自己声明听牌;若广播他人听牌状态=公开元信息(N5 范畴),非手牌真值 |

### 1.3 唯一两个潜在真 oracle (R9 / R10) 的深度分析

**R9 (查叫) 和 R10 (一炮多响) 是仅有的"服务端校验逻辑必须读取对手隐藏手牌"的动作。** 这是诚实的结论:绝大多数状态机校验只读请求者自己的牌(R1-R8、R11 全部零价值),只有"局末多家判定"类逻辑服务端不得不展开所有人的手牌。

但它们落进 **H16 hard wall**,被三重证据墙挡死:

1. **D23 证伪** (`archive/.../06-19-h16-bypass-parent/research/d23-premise-refuted-and-h16-reassessment.md`):没有任何"局末摊牌帧"存在。26K 完整金币局样本打到 win,对手手牌仍是 0x3c 占位。该游戏**没有"局末亮对手牌"这种 UI 帧**。
2. **win 帧 (0x0220) 实证 = 23B 纯数值结算摘要** (`08000000...41010300...`),无手牌 tile 数据。服务端判定一炮多响/查叫的**结果**(谁赢、退多少税)写进 win 帧,但**判定所依据的对手手牌真值不下发**。
3. **D22 最强铁证**:round_start 名册帧明文下发对手昵称/头像/分数/IP,**唯独手牌字段精确替换为 0x3c**。H16 是序列化层有意设计的安全基线,不是漏配。

**关键 oracle 失败原因**:R9/R10 的服务端**确实**在内部读取了 H 来做判定,但它把判定**结果**(布尔/数值,如"对手听牌=true、退税=N")序列化下发,而**不把判定输入**(对手手牌)下发。错误差异只能泄漏"判定结果",不能反推出 H 的具体牌张。

→ 除非存在一个 bug:服务端在某种**非法局末序列(状态机重置)**下,重新序列化全桌时**漏掉了 0x3c 抹除步骤**,把判定输入(对手手牌)也下发了。**这正是 §3 live 验证要打的唯一靶点。**

### 1.4 §1 小结

- **R1-R8 + R11 (9 个动作): 零 oracle 价值。** 服务端只校验请求者自己已知的牌/状态。错误差异化反推不出对手手牌。
- **R9 + R10 (2 个动作): 唯一理论 oracle**,但服务端只下发判定结果不下发判定输入,被 H16 全局过滤挡死。
- **唯一活口**:非法动作触发**状态机重置/全桌重序列化**时,服务端是否漏抹 0x3c(序列化路径分叉 bug)。**纯离线无法判定,必须 live 验证。**

## 2. 可观测差异维度表

即使错误**内容**不泄漏 H,**时序/副作用**仍可能泄漏。每发一个 (动作 A, 当前状态 S) 要记录全部维度:

| 维度 | 取值 | 是否可能泄漏 H | 说明 |
|---|---|---|---|
| `flag` | SUCCESS=0 / SHOW_MESSAGE=1 (房间级) | 否 | 只反映你自己合法性 |
| `msgbox.text` | 错误文案字符串 | 几乎否 | 文案模板化("没轮到你"等);除非文案里嵌对手信息(未见) |
| `action_rsp (0x2C2F)` 局内拒绝码 | 静态未知,需 live 捕获枚举 | **待测** | 局内动作的拒绝结构静态缺失,这是 fuzz 要测绘的主要未知 |
| **响应延迟 Δt** | 毫秒 | **理论可能** | 服务端校验路径长短不同→延迟不同。但单机方差大,需大量重复+统计,信噪比极低。**降级为弱信号** |
| **是否状态重置** | 服务端重发完整状态帧? | **最有价值** | 用户原话"有的直接重置"。若重置触发全桌重序列化且漏抹 0x3c = bug。**§3 主靶** |
| **重置时下发的帧内容** | 重发的状态帧逐字节 | **决定性** | 重置后的全桌帧若含对手 13B 真手牌块(非 0x3c/0x72)= oracle 命中 |
| 是否断连 (FIN/RST) | 连接是否被踢 | 否 (但是风控信号) | n6 实证:单连接≥6 不同 sub_type 触发断连墙。断连=风控,提示封号风险升高 |
| 是否触发反作弊 flag | 后续登录被标记/封号 | 否 | **高危副作用**。发非法局内动作是反作弊重点监控项 |
| 是否触发重发当前状态帧 | 服务端补发同步帧 | **可能** | 同"状态重置";要逐字节看补发帧是否漏抹 |

**结论**:在所有维度里,**唯一有 oracle 潜力的是"状态重置/重发帧的内容"**——其它维度要么零价值(flag/msgbox 反映自己),要么弱信号(延迟)。

## 3. Live Fuzz 测试方案 (设计,不执行;需用户副号 + 显式确认)

> ⚠⚠⚠ **此方案需用户用副号在 live 对局执行,有封号风险。发非法局内动作正是反作弊重点监控项。主号是所有攻面的命根子,严禁用主号。必须用户显式确认后才执行。** ⚠⚠⚠

### 3.1 方案目标 (收窄到唯一活口)

§1 已证 R1-R11 的**错误内容**都不泄漏 H。所以 live fuzz **不是**为了"看报错文案",而是为了验证**唯一活口**:

> **某个非法局内动作 → 触发服务端状态机重置/全桌重序列化 → 该重序列化路径是否漏抹对手手牌的 0x3c?**

观测重点 = **重置/重发帧的逐字节内容**,不是 flag/msgbox。

### 3.2 测试动作序列 (分风险等级)

#### Level 0 — 房间级 (低风险,可先做以校准观测管道)
- 错状态发 ReqPlayerAct:已 READY 又发 READY、已坐下发 SITDOWN、非法 sitorder CHANGESEAT、越界 tableID。
- **目的**:仅校准"如何捕获 RespPlayerAct flag/msgbox + 测响应延迟基线"。**明确:此层零 oracle 价值** (R1),只做管道验证。
- 风险:低 (房间级动作本就是客户端常发,错误只回 msgbox)。

#### Level 1 — 局内时序违规 (中风险)
- 不该你出牌时 discard (R2)、重复 discard (R4)、跳轮动作。
- **观测**:服务端是否回 action_rsp(0x2C2F) 拒绝码?是否**重发当前牌局状态帧**?若重发,**dump 整帧逐字节,搜对手位是否出现非 0x3c/非 0x72 的 13B 真手牌块**。
- 风险:中。时序违规可能被记为客户端 bug(正常网络抖动也会产生),但高频会触发风控。

#### Level 2 — 局内非法牌动作 (高风险,反作弊高危)
- 出不在手里的牌 (R3)、非法 chi/pon/kong (R5/R6)、非法 win (R7)。
- 制造**局末异常序列**:在临近流局/多家可胡时发越权胡/越权杠,试图触发查叫(R9)/一炮多响(R10)的重序列化路径。
- **观测**:同 Level 1,重点 dump 任何"状态重置/全桌重发"帧。
- 风险:**高**。这些是反作弊最敏感的项(出不在手里的牌 = 经典外挂特征)。**强烈建议仅在确认 Level 0/1 观测管道可靠、且用户明确授权后,极少量(<5 次)、低频(单连接≤3 异常帧,遵守 n6 断连墙)执行。**

### 3.3 观测与记录 schema

每个 (动作 A, 当前状态 S) 记一条:
```
{action, state_snapshot, sent_ts, flag, msgbox_text, action_rsp_code,
 resp_delay_ms, state_reset:bool, resync_frame_hex, opponent_hand_bytes,
 opponent_hand_is_0x3c:bool, disconnected:bool}
```
**命中判据**:`opponent_hand_is_0x3c == false` 且 `opponent_hand_bytes` 是 13 个互异 instance id (条→万→筒→字 顺序,见 memory stable-022b-1v1-deal) → **真 oracle 命中,立即详记并停手**。

### 3.4 风险分级汇总

| Level | 动作类 | 封号风险 | oracle 潜力 | 建议 |
|---|---|---|---|---|
| 0 房间级 | ReqPlayerAct 错状态 | 低 | 零 (仅校准管道) | 可先做 |
| 1 局内时序 | 越权/重复 discard | 中 | 低-中 (看是否重发帧) | 谨慎,低频 |
| 2 局内非法牌 | 越权 chi/pon/kong/win,局末异常 | **高** | 仅此层有触发查叫/重序列化的可能 | **需用户显式授权,极少量** |

## 4. 复用 n6_fuzzer 框架的改造点

n6_fuzzer (`archive/.../06-19-n6-hidden-protocol-scan/research/n6_fuzzer.py`) 是**扫 msg_type 字典**(固定 body,变 mt/sub_type)。状态机 fuzz 是**固定 msg_type,变 body / 变时序 / 变当前状态**。复用与改造:

**可直接复用:**
- `SRSClient` 连接 + 握手 + AES-CFB128 fresh-from-IV 加密 (`_encrypt_body`, `send_frame`)
- 连接健康探测 `connection_alive()` (SO_ERROR + _running + _sock 三信号)
- 断连墙规避:`--sub-per-conn` / `--camouflage` 的"单连接≤N异常帧 + 重连冷却"机制 (n6 实证墙=单连接≥6 不同 sub_type)
- dry-run / `confirm_live()` 安全门 / 副号检查 / JSONL 记录管道

**必须新增/改造:**
1. **目标从 lobby(5748) 改为 game-process**。局内动作要先进牌桌:`_resolve_host_port` 的 `game` 分支 (47.96.0.227:5045);且必须先发 ReqJoinRoom(11007) + ReqPlayerAct(SITDOWN/READY) 真正坐进一局,才能发局内 0x2BC0 动作。**这是 n6 没有的"前置进桌状态机"**,要新写。
2. **body 构造从 msg_type 字典扫改为"固定动作 + 变体 body"**:
   - discard body: 变出牌 tile id(在手里的合法张 vs 不在手里的非法张 vs 越权时机)
   - meld body: 0x021F 结构 (见 protocol.py `_extract_meld_info` L668-690,meld_count + tiles)
   - 用 `BODY_TEMPLATES` 机制扩展,新增 `discard_legal/discard_illegal/meld_illegal/win_illegal` 模板。
3. **响应分类器改造**:`classify_response` 现在按 mt 打分找 zlib/13B。状态机 fuzz 要新增:
   - 捕获 `action_rsp(0x2C2F)` 拒绝码(静态未知,需 live 枚举建表)
   - **检测"状态重置/重发帧"**:收到 round_start/hand_update/deal 重发时,**逐字节扫对手位是否非 0x3c**(复用 D22 的 0x3c 检测逻辑)。这是 oracle 命中判据,必须新写。
   - 响应延迟 Δt 记录(发送→收到对应 action_rsp 的毫秒差)。
4. **状态跟踪**:需维护"当前轮到谁/我手里有什么牌"以构造"越权/非法"动作。可复用 `stable/tracker.py` 的 `PacketStateTracker`(从 game-event 重建 BattleState),让 fuzzer 知道当前合法动作集合(也能从 action_notify 0x0016 直接读)。

**改造工作量评估**:前置进桌状态机 + body 模板 + 重发帧检测分类器是三块新代码;连接/加密/安全门/断连规避全部复用。约 60% 复用。

## 5. 诚实可达性结论

### 5.1 这条路是死、是活、还是需 live 验证?

**分层结论:**

1. **错误内容差异 (flag/msgbox/拒绝码) 这条路 = 死。**
   §1 判定表证明:R1-R8 + R11 共 9 个动作的服务端校验**只读请求者自己已知的牌/状态**,错误差异反推不出对手手牌。且 `action_notify(0x0016)` 服务端**主动预告**你的合法动作集,连"靠报错探测自己能不能吃/碰/胡"都不需要——服务端白送。这层**零 oracle 价值,确定无疑**。

2. **唯二理论 oracle (R9 查叫 / R10 一炮多响) = 服务端读了 H 但不下发 H。**
   这两个局末多家判定逻辑服务端**确实**必须展开所有人手牌,但它只序列化**判定结果**(谁赢/退税数值,见 win 帧 23B 实证),**不序列化判定输入**(对手手牌)。被 H16 序列化层全局过滤(D22/D23 三重铁证)挡死。**理论 oracle 存在,但泄漏不出牌张。**

3. **唯一未排除的活口 = 状态机重置时的序列化路径分叉 bug。** **需 live 验证,纯离线无法判定。**
   用户原话"有的状态机直接重置"指向唯一可能:某个非法局内动作触发服务端**重发全桌状态/重序列化**,若该重序列化代码路径**与正常路径不同**且**漏掉了 0x3c 抹除步骤**,就会把对手手牌真值下发。这是一个**实现 bug 假说**,不是协议设计漏洞。
   - 概率评估:**低**。D22 证明 0x3c 抹除是序列化层有意基线;但"异常/重置路径"是否走同一序列化函数,静态包(热更牌局场景缺失)看不到,只能 live 打出来看。
   - 这是 §3 Level 1/2 live fuzz 的**唯一靶点**:不为看报错,只为捕获"重置/重发帧",逐字节验对手位是否漏抹 0x3c。

### 5.2 是否发现某动作的错误差异确实依赖对手隐藏牌(真 oracle)?

- **协议设计层面**:R9(查叫)/R10(一炮多响)的**校验逻辑**依赖对手隐藏牌,是仅有的两个"校验必读 H"的动作。
- **但可观测的响应**:不依赖 H 的具体牌张(只下发判定结果)。所以**不构成可利用的 oracle**——除非 §5.1(3) 的重置序列化 bug 成立。
- **诚实判定**:**没有任何一个动作,其正常错误响应的内容直接是对手手牌的函数。** 唯一活口是实现 bug 假说,需 live 验证才能证实或排除。

### 5.3 一句话结论

> **状态机错误差异 oracle 这条路:错误内容层面已死(所有校验只读自己牌 + 服务端 action_notify 白送合法动作);唯二理论 oracle(查叫/一炮多响)读了对手牌但只下发判定结果不下发牌张,被 H16 挡死;唯一未排除的活口是"非法动作触发状态机重置时,重序列化路径是否漏抹 0x3c"——这是一个低概率实现 bug 假说,纯离线不可判定,需用户副号在 live 对局极少量、低频、显式授权下执行 §3 方案,捕获并逐字节验证重发帧来证实或排除。**

## Caveats / Not Found

- **局内动作响应结构 (action_rsp 0x2C2F) 静态缺失**:在热更牌局场景模块,本地解密 lua 包没有。其拒绝码枚举只能 live 捕获建表。这是 fuzz 要测绘的主要未知。
- **响应延迟 oracle 信噪比未实测**:理论上校验路径长短→延迟不同,但单机网络方差大,需大量重复+统计。本文降级为弱信号,未设计统计方案。
- **状态机重置是否走独立序列化路径 = 无法静态判定**:决定唯一活口成败的关键,静态包看不到,必须 live。
