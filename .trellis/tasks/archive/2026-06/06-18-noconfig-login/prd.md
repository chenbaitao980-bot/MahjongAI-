# brainstorm: noconfig login + credential persistence

## Goal

Add a real login step for noconfig admin entry so users without an existing account/password flow can still authenticate before entering the page, and support remembering login credentials so they do not need to sign in every time.

## What I already know

* The current noconfig admin entry is `GET /admin` in `remote/noconfig/app.py`.
* Current protection is a shared `api_token`, not an account/password login.
* The admin page currently reads token from `?token=` first, then `localStorage.mj_admin_token`, and uses it for `/api/users`, `/api/users/search`, and the hand iframe.
* Backend APIs such as `/state`, `/register`, `/push`, `/presence`, `/watch-info`, `/api/users`, `/api/users/search`, `/api/users/{user_id}` all validate `_check_api_token(...)`.
* The repo does not appear to have an existing session/cookie auth pattern in `remote/`.
* This repo already has product pressure around noconfig multi-user admin, user search, online visibility, and live hand parity, so the login flow needs to avoid breaking those existing admin workflows.
* Existing repo constraints show noconfig runs on port `8002`, and `/static/index.html` also accepts token via URL/local storage.

## Assumptions (temporary)

* This login is only for the `/admin` web UI entry, not for extractor/relay machine-to-machine callbacks such as `/push` or `/presence`.
* We can introduce a dedicated admin auth layer without changing the existing game-side credential capture model.
* "Remember login" should avoid storing raw password in browser storage.
* For MVP, a single admin account may be acceptable unless the user explicitly wants multiple admin operators.

## Open Questions

* How should the first admin account be provisioned: config file or first-visit bootstrap?

## Requirements (evolving)

* Visiting the noconfig admin page should require login before showing the admin UI.
* The login form should use account + password, not manual token entry.
* The system should support a remembered login state so users do not need to sign in every visit.
* The scope of this task is limited to `http://8.136.37.136:8002/admin`.
* Existing multi-user admin capabilities should remain usable after login.
* Existing extractor / ECS / proxy callbacks and non-admin routes should remain unchanged.

## Acceptance Criteria (evolving)

* [ ] Unauthenticated visits to `/admin` are redirected to or shown a login experience.
* [ ] Valid account/password grants access to the admin page.
* [ ] A remember-login option keeps the user signed in across browser restarts or revisits.
* [ ] Logging out removes remembered access.
* [ ] Existing admin list and hand-view flows still work after authentication.
* [ ] Existing non-admin token-based machine flows still behave as before.

## Out of Scope (explicit)

* Reworking spectator concurrency limits or `8003` single-bind architecture
* Reworking extractor credential capture, SRS session handling, or presence routing
* Reworking general API auth outside the `/admin` entry and its immediate admin-page needs
