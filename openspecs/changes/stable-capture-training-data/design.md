# 设计：stable-capture-training-data

## 当前状态

已有能力：

- 稳定抓包读取器可以输出结构化事件和 `PacketStateTracker.snapshot()`。
- 稳定版硬算面板已经约束为只使用同一份 `snapshot()` 数据。
- `shanten` 和 `ukeire` 可提供可验证的局面特征。
- `ExcelGameLogger` 可用于人工复盘和纠错。
- `mc_simulator` 和 `mc_policy` 可作为后续自博弈入口。

主要缺口：

- 缺少面向训练的数据集格式。
- 缺少从抓包日志回放到训练样本的稳定导出流程。
- 缺少动作标签可信门槛。
- 完整计分、包牌、对手模型尚未闭环，不能直接训练最终策略网络。

## 方案

分三阶段推进：

### 阶段 1：抓包稳定版数据闭环

把稳定抓包视为唯一事实源：

```text
抓包事件
  -> MJProtocol 结构化事件
  -> PacketStateTracker.apply()
  -> PacketStateTracker.snapshot()
  -> 可信门槛检查
  -> 训练样本 JSONL
  -> 人工纠错/回放验证
```

训练样本只记录可验证事实：

- 局面 ID、事件序号、时间戳。
- 我方手牌、副露、弃牌。
- 对方可见副露、弃牌。
- 财神、剩余牌数、当前回合。
- 可选动作、真实动作、动作牌。
- 当前向听、打后向听、有效进张。
- 数据可信状态和阻断原因。

### 阶段 2：监督学习

先用抓包日志学习真实玩家动作分布：

- 输入：可信局面特征。
- 标签：真实动作。
- 输出：本地候选重排模型，只允许重排硬算合法候选，不允许生成额外动作。
- 验证：离线 top-k 命中率、退听率、打财神率、未知数据阻断率。

### 阶段 3：自博弈强化

在以下前置条件满足后再做：

- 完整计分可用于 reward。
- 包牌/不死包规则可用于终局责任归因。
- 对手模型至少能从抓包监督数据初始化。
- 自博弈环境与稳定抓包事件格式可互相回放验证。

## 边界

- 本 change 不实现训练代码。
- 本 change 不修改 `mc_simulator`、`mc_policy`、`shanten`、`ukeire`。
- 本 change 不把 AI/LLM 输出当作训练标签。
- 本 change 不要求一次性完成完整计分和包牌。

## 回滚方案

删除 `openspecs/changes/stable-capture-training-data/` 规划目录即可，不影响运行代码。
