# stable-reader-optional-action-round2-fixes

## 为什么

用户在 2026-05-20 20:44-20:54 的实战抓包中发现稳定版面板仍有一组影响可用性的缺口：

1. 游戏画面出现“吃/过”“碰/过”时，抓包 UI 没有展示这类“可选动作”。
2. 第二张图中，真实手牌包含三万、三万、二条、七筒、九筒、南、南、九条，但面板缺少九条。
3. 敌方回合时，策略建议和候选分析仍显示旧建议或保守分析，应临时关闭或显示“等待敌方出牌”。
4. 第二把开始持续提示“大模型输出不合法”，没有稳定给出出牌建议。
5. 杠后真实手牌为一万、三万、九万、中、五万，但 UI 多出一个西；副露区中西西西、二筒一筒三筒、九条九条九条是对的。
6. 头上有三角形标记的牌需要在 UI/日志中备注，例如 20:52:04 的五万和一条、20:54:29 的七万和一条。

## 预检证据

- `data/stable_reader/events_20260520_204016.jsonl` 中 `action_notify` 出现 668 次，但 `stable/protocol.py` 目前只给 `0x0016` 命名，没有解码为可选动作结构。
- `data/stable_logs/牌面流水_20260520_204202.xlsx` 在 20:46:01 至 20:46:30 记录的我方手牌包含“三万 三万 二条 七筒 九筒 南 南”，没有九条；同一局 20:46 附近没有新的可信 13/14 张全量 `hand_update`，因此 UI 依赖增量事件维护，容易在吃碰/可选动作窗口后少牌。
- `logs/mahjongai_20260520_204007.log` 在 20:51:06 记录 `meld 0x021F not recognized, body_hex=00050104434343430343434300000000`；同一时刻 Excel 显示杠后手牌仍带“西”，说明杠牌识别/移除链路不完整。
- `events_20260520_204016.jsonl` 在 20:52:47 出现 `0x0216` 大体量副露汇总包，被当前代码按 `hand_update` 处理成 `hand_raw=[1,4,67,67,67]`，这是“多西/手牌被副露包污染”的直接风险点。
- `events_20260520_204016.jsonl` 在 20:53:49 记录 `body_hex=01010003375339015300000000`，包含财神替代值 `0x53` 的副露未能识别，说明副露解析不能只接受天然顺子/刻子。

## 影响面

| 文件 | 预计改动 |
| --- | --- |
| `stable/protocol.py` | 解码可选动作 `action_notify`；识别副露汇总型 `0x0216`；修正杠/财神替代副露解析 |
| `stable/tracker.py` | 维护可选动作状态；敌方回合阻断分析；杠后从手牌移除正确数量；保留三角标记备注 |
| `ui/stable_battle_panel.py` | 展示可选动作、敌方回合等待态、三角标记备注；清空过期策略/候选表 |
| `game/llm_advisor.py` | 大模型输出不合法时更稳地归一化/兜底，确保仍给程序建议 |
| `tests/test_stable_reader.py` | 增加上述协议与状态回归测试 |

## 验收

- [ ] 画面出现“吃/过”“碰/过”时，面板可展示“可选动作：吃/碰/过”等内容。
- [ ] 20:46 场景不因可选动作窗口或缺少全量手牌包丢失九条；无法确认时 UI 显示“不完整/等待可信手牌”而不是静默少牌。
- [ ] 当前回合为敌方时，策略建议和候选分析显示等待态，不复用上一条建议。
- [ ] 第二把开始 LLM 输出不合法时仍回退到程序候选，且原因说明可读。
- [ ] 20:51 杠后手牌不再多出西；副露仍展示西西西、二筒一筒三筒、九条九条九条。
- [ ] 头上三角标记的牌进入 UI/Excel 备注，例如“标记牌：五万、一条”。
