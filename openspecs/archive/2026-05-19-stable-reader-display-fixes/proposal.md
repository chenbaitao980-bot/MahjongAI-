# 稳定版抓包显示修复

## 为什么要做

实测对局中发现两个问题：

1. **手牌顺序与游戏 UI 不一致**：协议解码出的 13 张手牌牌值正确（多重集与实际相同），但顺序不是游戏 UI 上的排序。例如实际手牌 `四万 一条 一条 五条 一筒 五筒 六筒 六筒 七筒 八筒 九筒 九筒 白`，UI 显示成 `五筒 一筒 五条 九筒 白 六筒 九筒 一条 六筒 八筒 一条 七筒 四万`，玩家根本无法对照游戏画面校验。

2. **弃牌区偶尔出现幽灵牌**：实际弃牌序列 `东 二筒 一万 四条 九条 南`，识别出 `东 二筒 一万 四条 四万 九条 南`，中间多了一张本应在手牌中的 `四万`。npcap 在链路层抓到 TCP 重传包，应用层重复解码同一帧，导致 discard 事件被处理两次。

## 变更内容

1. **手牌排序**：在 `PacketStateTracker.snapshot()` 输出前按 **万→条→筒→字** 顺序排序手牌（与游戏 UI 一致）。内部状态保持协议原序（不影响事件解码）。
2. **TCP 序号去重**：在 `_parse_ip_tcp_static` 提取 TCP `seq` 字段，`MJProtocol` 维护每个 TCP 流的 `next_expected_seq`，重传包的字节不再喂入帧缓冲。

## 不在范围内

- 不改协议解码逻辑（msg_type / sub_cmd 解析、字段含义不变）。
- 不改视觉识别（HOG 流水线）。
- 不改对面/旁家手牌（始终隐藏，无排序需求）。

## 成功标准

- 我方手牌 UI 显示顺序与游戏画面完全一致（万 在前，条 次之，筒 再次，字 最后；同花色按数字升序）。
- 整局对局中弃牌序列不再出现幽灵牌（与游戏画面 1:1 对照）。
- tcpdump 模式不受影响（同一份 protocol/tracker 代码两种抓包模式共用）。
- 回放脚本 `scripts/replay_stable_reader.py` 仍可正常运行。
