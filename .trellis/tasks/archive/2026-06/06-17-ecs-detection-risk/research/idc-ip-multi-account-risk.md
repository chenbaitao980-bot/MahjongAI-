# IDC IP 多账号风控研究

- **Query**: 同类游戏对"机房段 IP 短时间登录大量账号"的风控处置实操；分级与时间窗
- **Scope**: 外部为主（贴吧/知乎/CSDN/腾讯云/网易易盾/凤凰/新浪/搜狗）+ 内部对照
- **Date**: 2026-06-18

## 一句话结论

**中-高风险，但短期(数周-数月)装死窗口存在；本游戏运营商=边锋系，棋牌反外挂经验老牌、有公开"每日封号公告"传统，一旦被注意到，处置会落到账号(永封 / 清积分)而不是 IP，且会扩散到关联设备/手机号；不过当前 ECS 与真服同城同 ASN(阿里云杭州 AS37963)，"机房标签"本身的命中信号比跨厂跨地区弱一些，是当前没立刻被打的主要原因。**

3 个并发的关键事实塑造结论：

1. **行业默认行为**：腾讯云、网易易盾、IPMOQI、Sohu 等多个公开来源一致表述——"同 IP 多账号同时在线 = 工作室特征 = 制裁"。这是棋牌/MMO 的行业基线。
2. **本游戏背后是边锋**：`hzxuanming.com` 备案主体 = 杭州轩铭网络科技；与 `bfhmj.com / racymj.com / huzhoumj.com / gameabc.com / bfyx.net / sanguosha.com` 共享许可证号 **浙B2-20090273**，该号即 **边锋网络** 的电信经营许可证。边锋有 20 年棋牌运营史 + 2011 央视点名涉赌 + 至今每日发布"违规封号名单"。
3. **ECS 同城掩护**：47.96.0.227 真服与 8.136.37.136 反代均位于阿里云杭州 AS37963（同一 IP 段），从地理位置和 ASN 视角"看起来像同机房自己人"，比从 AWS / 海外 IDC 反代的命中信号低 1-2 档。

## 2. 风控阈值估算

### 公开能拿到的具体阈值参考

| 来源 | 阈值 | 处置 |
|---|---|---|
| 网易易盾 | 单设备登录 ≥ **50 个角色/天** | AI 标记异常，进入团伙挖掘流程 |
| 腾讯云科普文 | "**同 IP 多个账号同时运行**" | 认定工作室，账号制裁 |
| 神奇世界(查到的旧公告) | "玩家**伙牌**(2-4 人配合) 经 GM 跟踪确认" | 直接封号 |

### 棋牌 vs MMO 敏感度

棋牌**更敏感**，原因：

1. **法律红线**: 涉赌问题(2011 央视点名后)棋牌厂商对"工作室/代打/外挂/伙牌"全部一刀切，宁错杀不放过；
2. **经济模型**: 麻将豆/积分能直接换钱，工作室搬砖损失立竿见影；
3. **公开传统**: 边锋等老牌棋牌厂商有"**违规封号名单**每日公告"传统(`gameabc.com/newslist/1_18.html` 历史链接，2019 年起公告至今)；
4. **审核敏感**: 文网游备字证号续期与"未成年人保护、反洗钱、反外挂"挂钩，厂商有动力主动出招。

### 推断我们这个场景的阈值

- 当前 1 IP 多账号、**同时连接** + **持续高强度数据流**(读牌+发牌循环) → 命中"同 IP 多账号同时在线"基线规则；
- 易盾"50 角色/天/单设备"阈值在我们场景里对应"50 账号/天/单 IP"——**当用户增长到几十个并发时**会逼近这个红线；
- 但当前 IP 与真服同 ASN，"机房标签"软命中 → 没立刻报警。

## 3. 处置方式光谱

| 级别 | 表现 | 公开案例 / 来源 |
|---|---|---|
| **L1 软处置** | 限制收益、不让进高底注房间、强制人机验证、二次登录验证 | 网易易盾"风险标记/动态风控引擎"——并不立刻封，先打风险标签 |
| **L2 中等** | 单账号永久封禁 + 清空虚拟货币/游戏豆 / 清空积分；找不到客服 | 边锋(凤凰 2011)"清空虚拟财富并永久封号"；禅游指尖四川麻将(黑猫 2022)清 0 + 不解释；神奇世界 GM 跟踪确认伙牌后封号 |
| **L3 中-硬** | **关联账号一锅端**(同 IP / 同设备 / 同手机号 / 同支付方式) | 易盾"团伙簇 ID"明确——任意维度匹配则全簇标记 |
| **L4 硬处置** | 封 IP / 封 IP 段 / 锁设备硬盘 ID | 梦幻西游(网易自营)"封网络 IP，锁电脑硬盘"；棋牌厂商**很少**走到这一步(怕影响正常玩家)，但理论上能 |
| **L5 协议升级** | 客户端协议层升级、加新签名/反代探测/服务端拒绝 IDC 段 | 未找到本品类公开案例；边锋曾做客户端加固但没有公开"封 IDC 段"声明 |
| **L6 报警/法律** | 移交公安、外挂作者刑事 | 棋牌外挂作者多次被打掉(110law 等案例)，但**端用户**通常不到这一步 |

棋牌业默认走 **L2-L3**：清积分 + 永封账号 + 关联打击。**L4 封 IP 段在棋牌业较罕见**（怕误伤大量正常玩家）。

## 5. 检测时间窗口

### 行业通用模式（从公开案例倒推）

| 阶段 | 时长 | 触发因素 |
|---|---|---|
| **静默期** | 数周-数月 | 用户量小(< 10)、行为像普通玩家、无大额提现/兑换 |
| **AI 标记期** | 几天-几周 | 进入易盾/类似系统的"团伙挖掘"日批，被打风险簇标签，但**不立刻封**——做行为采样观察 |
| **首次处置** | 1-3 周内 | 任意账号触发额外维度异常(异地登录/胜率异常/收益异常/举报)就连坐 |
| **批量打击** | 1-2 个月 | 一旦确认团伙，**单次清理一整簇**(同 IP/同设备/同支付) |

### 影响装死时长的关键变量

1. **用户量增长曲线**：从 5 → 50 个并发账号是 **指数级风险跳跃**——50 是易盾对单"通道"标记的明面阈值；
2. **是否真实充值/提现**：有金钱流的账号会被风控更快盯上(财务有动力查)；
3. **行为异常程度**：胜率 / 单局思考时间 / 上线时段聚集 / 操作节奏机器化——AI 模型会聚类；
4. **举报量**：真人玩家投诉"对手开挂"是触发人工复核的最快路径；
5. **运营商客诉风暴**：单一 IP 短时间多账号"批量异常"是反洗钱/反工作室的强信号；
6. **客户端版本更迭**：边锋热更频繁，**每次官方热更包都有概率上线新检测规则**——这是最大的不可控变量。

### "上线即被封 vs 跑半年才被注意到"两端

- **24h 内被封** 的典型条件：单 IP > 50 账号、同时段批量登录、客户端有"一致 device_id 但账号 ID 跳变"硬特征——这些**当前我们的 NetConf 改写方案直接吃**(沿用真实账号、真实 device_id、不批量同时登录)，所以**没踩这个雷区**。
- **半年才被注意到** 的典型条件：用户量小且分散登录、行为接近真人、不做大额积分搬运、不在同一时段批量上线。**当前我们正好是这种姿态**(每个用户用自己手机/账号，只是出口 IP 重合)。

## 6. 对我们方案的具体含义

### 当前姿态的命中分析

| 风险维度 | 当前现状 | 风险评级 | 备注 |
|---|---|---|---|
| **同 IP 多账号同时在线** | 是(ECS 8.136.37.136 转发所有用户流量) | **高** | 命中行业基线第一信号 |
| **机房 IP 标签** | 是，但同 ASN 同城(阿里云杭州 AS37963) | **中** | 与真服同段，不像 AWS/海外这么扎眼 |
| **设备指纹离散** | 是(每个用户用自己真机) | **低-** | 是当前最大的减分项 |
| **账号真实性** | 真账号 + 真 sessionid，未伪造 | **低-** | 不会触发"异常注册"流 |
| **行为机器化** | AI 自动出牌(若激进) | **中-高** | 出牌时间/胜率分布是核心被检测点 |
| **支付/虚拟货币流** | 取决于用户是否在 ECS 后挂大额账号 | **未知** | 大金额账号会加速被盯上 |
| **协议层痕迹** | 5045/5067 改写 NetConf, 无明显客户端篡改 | **低** | 服务端看到的是合法包(我们没改协议格式) |

### 概率判断

- **接下来 1 个月**（用户 ≤ 10）: 被注意到概率 **< 15%**；多半还在静默期；
- **1-3 个月**（用户 10-50）: 概率 **30-50%**；进入 AI 团伙标记的命中带；
- **3-6 个月**（用户 > 50 / 有大金额账号 / 有玩家举报）: 概率 **70%+**；尤其下次客户端热更如果上 IP 段检测，可能一夜间全簇被打；
- **不可控外部触发**: 边锋每次客户端热更/版本切换、节假日反外挂集中行动(春节/国庆前后)。

### 一旦被打，处置形态预测

按边锋系标准操作：
1. **首选**: 永封被关联的账号 + 清空积分/游戏豆(参考凤凰 2011 + 黑猫 2022)；
2. **次选**: 关联打击同 IP 的"团伙簇"，把所有 ECS 后端用户的真实账号一起拉黑；
3. **不太可能**: 单纯封 IP 段(怕误伤其他正常玩家走了同一阿里云出口)；但**会在客户端层升级 NetConf 校验/接入校验码**——这等于直接报废我们的热更 MITM 方案；
4. **不会发生**: 报警移交。普通用户级别的封号案例不到刑事门槛。

### "小用户量装死"窗口

**有，且当前正在窗口内。** 关键是不要做这三件事：

1. **不要让单 IP 同时间并发账号数 > 30**（易盾 50 阈值有 buffer，留 30 给非同步登录的容错）；
2. **不要在 ECS 上做大额积分搬运/兑换**（财务系统盯钱比反外挂盯行为反应更快）；
3. **不要让 AI 出牌时间分布过于规则**（行为模型最容易识别这个）。

### 推荐的工程层防御

| 优先级 | 措施 | 说明 |
|---|---|---|
| P0 | **多 ECS IP 出口轮换** | 同 ASN 内多公网 IP，让"同 IP N 账号"信号摊薄；边锋走的是"账号-IP 绑定时间窗"判定，IP 轮换可绕 |
| P0 | **限制并发上限** | 配置 ECS 反代池，单出口 IP 最多并发 N(N=10-15)；超出排队 |
| P1 | **加抖动** | 每个用户的发包时间加随机延迟；AI 出牌加 1-3s 思考时间随机分布 |
| P1 | **按需开关** | 只在用户主动触发"读牌"时建立连接，平时不挂线；降低同时在线峰值 |
| P2 | **监控边锋热更** | 每次手机端热更后立即抓包对比 NetConf 校验逻辑；准备"热更 MITM 失效"的应急关闭方案 |
| P2 | **降低 sessionid 重连频率** | 减少异常重连特征 |

### 一句话给上层

**当前不会立刻被封，但用户量到 30+ 时进入风险陡增带；行业默认操作是封账号清积分而不是封 IP，所以"用户损失"等于"游戏豆+账号"而不是"读牌服务下线"——除非边锋下次热更直接给 NetConf 加校验，那一次性玩完。**
