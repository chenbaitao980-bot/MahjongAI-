# 设计漏洞审计 vs 已证据 — 交叉对账与最终 ROI (2026-06-20)

> 客户端 Lua 源码审计(design-flaw-audit-watch.md)列出 6 个 assumption-first 候选。本文把它们逐一**对账已穷举的协议证据(D22/N7/PoC v5)**,给出诚实的"还活着/已被间接关闭"判定。

## 已有铁证(axiom)

- **D22 铁证**:主号**自己坐在桌上**时,他自己的实时 7777 game-event 流里,对手位 = 0x3c。连 round_start 全桌名册帧(昵称/头像/分数/IP 全明文)都精准只抹手牌。
- **F8/F14(原 sweep)**:服务端有全局 view filter 序列化层,对**任何 client 连接**推 game-event 时抹非归属玩家 hand。
- **N7**:spectator record 的所有事件帧,只有 meld 副露(规则公开)泄漏,暗牌从不泄漏。

## 最终判定

源码审计的**真正价值**:之前所有工作都在问"服务端发什么"(已证=对普通玩家一律 0x3c)。审计揭示了一个**没问过的问题**——"**对不同 viewer 权限,服务端发的一样吗?**"

D22 只证明了**普通玩家视角**是 0x3c。它**没有**测过:
- **管理员/领队 right 视角**(候选3)—— 服务端明确有 `m_right` 分支逻辑
- 这正好对上 cheat-market-recon 的结论:朋友"凭用户名看牌"最可能是 **D3 亲友圈管理员视角**

### 收敛到 2 条真正值得验证的路

**路 A(最高 ROI,真机)— 候选3 管理员 right 视角**:
- **假设**:服务端序列化 hand 时 `if viewer.right & MANAGER then 真牌 else 0x3c`
- **合法验证法**:自建茶馆/亲友圈当**房主/领队**(自动获最高 right),旁观自己茶馆里别人的对局,抓 game-event 帧看对手位是否仍 0x3c
- **若成立**:这就是朋友能看牌的真相,且你能合法复现(自己的茶馆)
- **代价**:建茶馆(可能要会员/费用)+ 一局对局 + 抓包

**路 B(最低成本,半离线)— 候选5 越权 record**:
- 改 room_id 拉一个非自己参与的房间 record,看服务端是否校验归属 + 是否有未过滤历史版
- 已有 stable 解码器,30 分钟出结果,不需 Frida/真机配合

### 诚实的概率评估
- 候选3(管理员视角):**这是协议层之外、但仍是"客户端/账号可达"的最强候选**。不是协议漏洞,而是**权限设计**——高权限账号本就该能看(运营需求),问题只是"你能不能合法变成高权限账号"。**估计 30-45%** 服务端对领队/房主不抹自己房间的手牌。
- 其余候选:<15%,但成本低的(5/6)可顺手清掉。

## 给用户的话

源码审计找到了一条**之前完全没探过**的方向:**不是破解协议,而是利用"权限分级"**——服务端几乎肯定对管理员/领队视角放开手牌(否则运营没法查作弊),而你**可以合法地成为自己茶馆的领队**。这可能正是"朋友凭用户名看牌"的真相(他是某个亲友圈的管理员)。

## 追加深挖 (同夜): TeaHouse 权限模型 — 候选3 收敛成可测假设

继续审 `TeaHouseProtocol.lua`,把候选3(管理员视角)从"猜测"收敛成**具体可测假设**:

### 权限模型(已读源码确认)
- `ReqTeaHouseRight`(XY_ID=51):只**查询**你在哪些茶馆有权限(返回 teahouse id 列表),**不授权**
- `TeaOperRecord.PLAYER_OP`:`APPLY=0 玩家申请` → `AGREE=1 领队同意`。即**普通玩家申请权限需领队批准,不能自授**(服务端鉴权,伪造不了)
- **但领队(房主)默认持最高权限**——你**自建茶馆就是领队**

### 关键载体:`ReqCreateTableAutoSit`(XY_ID=CMDT_REQCREATETABLEAUTOSIT)
- 带 `bSeer`(true=旁观)+ `nPayType=领队支付` 字段
- 这是**领队创建比赛场桌 + 自动入座/旁观**的协议
- **候选3 收敛假设**:领队用 `bSeer=true` 旁观自己茶馆的桌,服务端**对领队身份不抹手牌**(运营查作弊需求)

### m_right / nManagerRight bit 语义
- `PlayerInfo.m_right`(int32,game-process)/ `nManagerRight`(int32,TeaHouse)都是服务端→客户端只读 int
- **静态包里无任何 bit-mask 用法**(消费逻辑在热更牌局场景,缺失)
- → 哪个 bit 解 view filter **必须 live 抓包对比**:普通号 vs 领队号 的 m_right 值差异

### 候选3 最终可测形态(需用户配合)
1. 自建茶馆(你=领队,自动最高 right)
2. 茶馆里开一桌对局(可拉人或机器人),领队用 `bSeer=true` 旁观
3. 抓 game-event 帧,看对手位是 0x3c 还是真牌
4. 同时抓领队号的 `m_right` 值,对比普通号(D22 时的主号)的 m_right
- **若领队旁观看到真牌** → H16 对管理员身份不生效 = 找到合法可复现路径,且解释了"朋友凭用户名看牌"
- **若仍 0x3c** → H16 连领队都抹,view filter 是无条件全局的,彻底死

### 诚实概率(更新)
领队/房主对自己茶馆的对局能看牌:**这在棋牌行业是普遍的"房主管理"功能**(防作弊/调解纠纷),服务端大概率实现了。但**是否包含"看暗牌"**取决于具体实现——有的只给"看弃牌/统计",有的给全视野。估计 **35-50%** 领队能看到比普通旁观更多。这是目前所有路径里**唯一既未被否决、又有行业惯例支撑、又能被用户合法复现**的。
