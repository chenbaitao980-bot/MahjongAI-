# Research: RNG / Seed / Wall Reconstruction Attack-Surface Sweep (思路 #5)

- **Query**: 发牌 seed / 牌墙顺序 / 局号随机数是否在客户端收到的帧里出现，能否本地重算整副牌（含对手手牌）
- **Scope**: internal (offline analysis of own captured traffic + decompiled lua)
- **Date**: 2026-06-20
- **Method**: 纯离线，零网络。解析 `sample_record_26k.bin` + `sample_record_33k_before_round.bin` 的全部 2BC0 子帧，逐字段拆解 round_start(0x0004)/deal(0x0003)/draw(0x021a)/discard(0x021b)；交叉比对两局；grep 反编译 lua 找洗牌/seed/random 逻辑。

> **Assumption-first 立场**：这条路只在"客户端能拿到牌墙顺序或洗牌 seed"时成立。下面先假设它可能成立，逐项找证据，再给诚实判定。

---

## 1. 帧格式（修正版，重要）

任务 brief 里的 "12B header / 32B ascii header" 描述不准。实测记录格式是**变长 ASCII 数字头 + 紧凑 SRS 子帧**：

```
[ascii-digit run ...][sub_cmd:2B LE][len:2B LE][body:len bytes]
```

- round_start 真实位置：`@436` = `04 00`(sub=0x0004) `33 01`(len=0x0133=307) 然后 body。
- 锚定子帧的可靠方法 = 扫描 `[已知sub_cmd LE][合理len LE]`（见 `parse_rng.py::walk`）。
- 26k 样本子帧统计：deal×5, round_start×14(其中 10 个是完整玩家块/4 个是 4 字节占位), draw×44, discard×47, meld×3, win×1, stat_update×190。

## 2. round_start (0x0004) 307字节 逐字段拆解

完整玩家信息块（注意：0x0004 出现多次，每次是**一名玩家的入局信息**，不是单一"局开始 seed 帧"）：

| offset | bytes | 字段语义 | 置信度 |
|---|---|---|---|
| +0 | `c5 1b 00 00` | areaid LE = 7109 (区号) | 高 |
| +4 | `43 95 8e 40` | u32，疑似账号/openid 句柄 (非 seed，见 §5) | 中 |
| +8 | `02 b7 e1 00` | u32 = 14792450 | 低 |
| +12 | `31 00 00 00` | u32 = 49（小整数，房间/座位相关） | 低 |
| +16..+47 | 全 `00` | 保留/对齐 | 高 |
| +48..+79 | `e4 03 .. / e8 03 ..` | **玩家分数/金币块**：0x03e4=996, 0x03e8=1000，4 个 8字节槽 | 高 |
| +87 | `04 00 00 00` | u32 = 4（玩家数 or 座位数） | 中 |
| +91 | `0d 00 00 00` | u32 = 13（**起手手牌张数=13**） | 中高 |
| +95 | `0a 00 00 00` | u32 = 10 | 低 |
| +103 | `03 00 00 00` | u32 = 3 | 低 |
| +111 | `91 1f b8 27` | **玩家 numid (4B)** = BE 0x911fb827 / LE 0x27b81f91 | 高 |
| +115 | `0d` + `39.184.31.145` | **玩家 IP**（1B 长度前缀字符串，len=13） | 高 |
| +130 | `00 00 02 00...63 00...` | 状态位 + 0x63=99(等级?) | 低 |
| +148 | `03` + `e4 b8 b0` | **昵称** "丰"（1B 长度前缀 UTF-8，len=3） | 高 |
| +152 | `9a` + `https://thirdwx.qlogo.cn/...` | **微信头像 URL**（1B 长度前缀，len=0x9a=154） | 高 |

第二个 round_start 块（@783, len=306）结构完全平行：areaid 7109 → 分数(996/1000) → numid `88 25 88 08` → IP "8.136.37.136" → nick "LOLLAPALOOZA"(len=0x0c=12) → 头像 URL。

**结论**：0x0004 = **玩家档案/入局信息广播**（区号+分数+numid+IP+昵称+头像）。`911fb827`/`88258808` 是**玩家账号 numid**，不是随机种子。整帧没有任何"牌墙/局号/seed"字段。`e4030000`/`0d000000`/`0a000000` 是**分数/手牌张数/计数**，不是 RNG。

## 3. deal (0x0003) 25字节 拆解

```
0b 0b 26 25 03 02 0e 20 07 12 13 14 15 | 0b 02 02 02 46 | 3c 3c 3c 3c 3c 3c 3c
[--------- self 13 张真值 instance-id --------]  [尾部5B]   [对手 7×0x3c 占位]
```

| 段 | 字节 | 语义 |
|---|---|---|
| +0..+12 | `0b 0b 26 25 03 02 0e 20 07 12 13 14 15` | **自己 13 张起手**（instance-id，0x3c 不在内 = 全部明文） |
| +13 | `0b` | 计数/分隔 = 11? (待定) |
| +14..+16 | `02 02 02` | 标志位 |
| +17 | `46` | 标志 (0x46=70) |
| +18..+24 | `3c × 7` | **对手 7 张手牌全部抹成 0x3c 占位** |

- 尾部 `0b 02 02 02 46` 不含任何 32/64-bit 高熵值 → **没有 seed，没有牌墙序列**。
- 两局样本 deal 的 self13 完全相同（`[11,11,38,37,3,2,14,32,7,18,19,20,21]`）—— 说明 33k 与 26k 是同一局/重叠抓包，不构成"两局不同 seed"对比，但足以证明 deal 帧里没有逐局变化的随机字段。

## 4. 牌墙可还原性判定（核心）—— **(d) 服务端洗牌，不可还原**

判定依据来自 draw/discard 帧的揭示规则：

**draw (0x021a, 7字节)**：`[player:1B][0x72 mask][tile:1B][00 00 00 00]`
```
00 72 00 00 00 00 00   <- 对手(player=0)摸牌，tile 字节恒为 0x00（永远不下发真值）
01 72 04 00 00 00 00   <- 自己(player=1)摸牌，tile=0x04（明文真值）
01 72 06 ...           <- 自己后续摸 06,08,0c,0f,11,13,15,17,19...（都是真值 instance-id）
```

**discard (0x021b, 6字节)**：`[player:1B][tile:1B][00 00 00 00]`
```
01 25  <- 自己弃 0x25（真值）
00 33  <- 对手(player=0)弃 0x33（真值！弃牌一律明文）
```

**关键事实链**：
1. 服务端对**对手摸进的每一张**都下发 `tile=0x00`（mask 0x72），从不给真值。
2. 服务端对**对手的起手 13 张**下发 0x3c 占位（deal 帧）。
3. 服务端**只在弃牌时**才揭示一张牌的真值（双方都揭示）。
4. **没有牌墙顺序数组**：draw 帧里没有"第 N 张牌墙位置"索引，对手摸牌位是纯 0x00 黑盒。
5. **没有 seed**：round_start/deal 全字段已拆，无任何逐局唯一的随机种子；§5 交叉比对也证伪。

这是**典型的服务端权威洗牌 + 按需揭示**模型：服务端在自己内存里洗好整副牌，发牌/摸牌时**只把当前玩家有权看到的那张牌的真值推下来**，对手位一律 mask。客户端从未收到牌墙序列或 seed，**没有任何信息可重算对手手牌**。

→ **判定 (d)：服务端洗牌，牌墙不可从客户端流量还原。思路 #5 这条路是死的。**

## 5. seed 字段证伪（双局交叉比对）

`verify_seed_hypothesis.py`：扫两局 round_start 全部高熵 u32（≥3 distinct bytes, >0x10000）。

- 26k 候选 457 个，33k 候选 510 个，两局共有 432 个。
- 共有值全部可归因为：**相同玩家账号 numid**（同账号在两局复现）、**相同分数常量**（0x3e4=996/0x3e8=1000）、**头像 URL 的 ASCII 字节滑窗**（如 0x00353431="541"）。
- **没有任何一个结构化的、逐局唯一的高熵值**符合"洗牌 seed"特征（seed 应只出现在某一局、且不是账号/分数/URL）。

→ seed 假设证伪：协议里不存在发牌随机种子。

## 6. 源码侧验证（§3 任务项）

| 检查 | 结果 |
|---|---|
| `apk_research/decrypted-lua` 全树 grep `math.random/randomseed/shuffle/洗牌/发牌/牌墙` | 86 命中，**全是非麻将逻辑**：网络重试(`NetEngine.lua:274`)、提示动画(`ScrollTipLayer.lua`)、SDK nonce(`dhsdk/Util.lua`)、埋点随机串(`ThinkData/Utils.lua`)、lobby 的"洗牌"小游戏(`ReqShuffle5.lua` — 是赌场翻牌活动，非麻将) |

| 麻将真正的发牌/round_start 解析代码 | **不在反编译包内**。包里只有 `games/GameHotUpdate*.lua`（热更下载器/资源校验），实际牌局模块是热更单独下载、本反编译快照缺失 |

## 7. 可达性结论

| 维度 | 结论 |
|---|---|
| **牌墙可还原性** | **(d) 服务端洗牌，不可还原**。客户端从不收到牌墙序列或 seed |
| **协议里有 seed 吗** | 无（§2 全字段拆 + §5 双局交叉比对双重证伪） |
| **客户端有洗牌代码吗** | 无（§6，发牌逻辑纯服务端/缺失模块） |
| **对手摸牌真值** | 永远 mask（draw tile=0x00 / deal=0x3c），与 H16 的 0x3c 墙同源 |
| **唯一揭示对手牌的合法时机** | 弃牌(discard 明文双方) + 局末摊牌(0x022B round_result，已在既往工作中确认会下发对手手牌 zlib 回放) |
| **思路 #5 总判定** | **死路**。服务端用"权威洗牌 + 按 view 权限揭示"模型，没有泄露任何随机性来源；无法靠分析自己流量重算对手手牌 |

### 与既往结论的关系（非新突破）
- 对手**最终手牌**仍只能从 **0x022B 局末摊牌**取得（既往 friend-watch-handcards-truth.md 已证服务端会在摊牌时发对手 hand 的 zlib 回放）。
- 对手**实时手牌**仍是 hard wall（H16 view filter 抹 0x3c），本次 RNG 角度**没有绕过它**。
- 唯一仍开放的实时读对手牌路径未变：手机进程内 Frida siphon（既往 siphon-final-goal.md）。

## Caveats / Not Found

- 33k 与 26k 样本 deal 完全相同（同一局/重叠抓包），所以"两局不同 seed"对比是**用 round_start 玩家块的高熵集合差异**代替做的（§5），而非用两副不同起手牌做的。若要 100% 钉死，可 live 抓**两局明确不同的对局**再跑 `verify_seed_hypothesis.py` —— 但当前证据（无 seed 字段、无牌墙数组、无客户端洗牌代码、对手摸牌恒 mask）已足以判定 (d)，**不建议为此额外抓包**，性价比低。
- 反编译包缺失麻将牌局模块，无法直接读到服务端发牌函数源码；但客户端协议行为（mask 规则）已是充分的间接证据。
- 既往 brief 里把 `911fb827` 当"可疑 32-bit seed"是误读 —— 实为玩家 numid（后面紧跟 1B 长度前缀的 IP 字符串，对齐完全吻合）。
